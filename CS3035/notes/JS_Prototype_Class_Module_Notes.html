01

Prototype Inheritance

JS

In JavaScript, objects can inherit properties from other objects via the prototype chain. Object.create(proto) creates a new object whose prototype is set to proto.

let honda = { make: "Honda" };
let civic = Object.create( honda );   // honda is civic's prototype
civic.model = "Civic";

console.log( civic );          // { model: "Civic" }  ← only own props shown
console.log( civic.make );    // "Honda"    ← inherited from honda
console.log( civic.model );   // "Civic"    ← own property
console.log( honda.model );   // undefined  ← honda has no model
Output Explained
civic only stores model — yet reading civic.make succeeds because JS walks up the prototype chain to honda. honda.model is undefined because inheritance is one-directional (child → parent, never parent → child).
02

Properties in Prototype Inheritance

JS
Reading a property

If the property doesn't exist on the object itself, JS walks up the prototype chain until it finds it or reaches null.

Writing a property

If the property doesn't exist on the object, a new own property is created directly on that object — the prototype is never touched.

Key Rule
Read → look up the chain. Write → always create/update on the current object only.
03

Inherited Methods

JS

Functions stored as properties are inherited just like any other property. Inside the inherited function, this refers to the object the method is called on — not the prototype it lives on.

honda.owner = "Car Dealer";
honda.printOwner = function() {
  console.log(this.owner);
};

honda.printOwner();   // "Car Dealer"   — this = honda
civic.printOwner();   // "Car Dealer"   — inherited; this = civic, civic.owner walks up to honda

civic.owner = "Chengyu";      // creates OWN property on civic
civic.printOwner();   // "Chengyu"      — this = civic, now has its own owner
this binding rule
this is always set to the object before the dot at the call site — civic.printOwner() → this is civic.

JS

Every JS object has an internal hidden slot called [[Prototype]]. It can be read/written via the legacy accessor __proto__ (or the cleaner Object.getPrototypeOf()).

05

What is prototype?

JS

Don't confuse [[Prototype]] (the hidden link between objects) with the .prototype property (a plain object that lives on constructor functions).

How it connects
When you call new Foo(), the newly created object's [[Prototype]] is set to Foo.prototype. This is how instance methods defined on Foo.prototype become available to all instances.
  • Object, Array, Function, etc. are all constructor functions with a .prototype object.
  • The "instance methods" you use daily (.push(), .map(), .toString()) are actually methods stored on those .prototype objects.
  • <Constructor>.prototype sets the prototype for every object that constructor creates.
06

Constructor Functions & Class Patterns

JS

Any function called with new becomes a constructor. Before ES6 classes, this was the standard way to create reusable object blueprints.

Pattern A — method on prototype (shared across all instances)

function Point(x, y) {
  this.x = x;
  this.y = y;
}
Point.prototype.toString = function() {
  return `(${this.x},${this.y})`;
};

var p = new Point(2, 3);
console.log(p.toString());   // "(2,3)"

Pattern B — method inside constructor (each instance gets its own copy)

function Point(x, y) {
  this.x = x;
  this.y = y;
  this.toString = function() {
    return `(${this.x},${this.y})`;
  };
}
Pattern A vs Pattern B — key difference
Prototype method (A): one function object shared by all instances — memory-efficient.

Constructor method (B): a new function object is created per instance — allows closures over private data, but uses more memory.
07

"Static" Members

JS

Before ES6 static, "static" members were simulated by adding properties directly to the constructor function — not to this or prototype.

function Point(x, y) { this.x = x; this.y = y; }

Point.origin = new Point(0, 0);    // "static" property
Point.distance = function(a, b) {   // "static" method
  return Math.hypot(b.x - a.x, b.y - a.y);
};

// Called on the constructor, not on instances
console.log(Point.origin);          // Point { x: 0, y: 0 }

08

ES6 Class Syntax

ES6+

ES6 introduced the class keyword — a cleaner, more readable syntax. Under the hood it is still constructor functions and prototypes; no new runtime mechanism was added.

class Animal {
  constructor(name) {
    this.name = name;       // instance property
  }

  eat() {                   // added to Animal.prototype
    console.log(`${this.name} is eating.`);
  }

  static describe() {      // static method on Animal itself
    console.log("Animals eat, sleep, and breathe.");
  }
}

let cat = new Animal("Whiskers");
cat.eat();           // "Whiskers is eating."
Animal.describe();   // static — called on class, not instance
Behind the scenes
class Animal { eat() {} } is functionally identical to:
function Animal(name) { this.name = name; }
Animal.prototype.eat = function() { … };

Inheritance & Instantiation

ES6+

Use extends to create a subclass and super() to call the parent constructor.

PARENT CLASS
Animal
+ name
+ eat()
↑ extends
Dog extends Animal
CHILD CLASS
Dog
+ name (via super)
+ breed
+ eat() (inherited)
+ bark()
class Dog extends Animal {
  constructor(name, breed) {
    super(name);             // must call super() before using this
    this.breed = breed;
  }

  bark() {
    console.log(`${this.name} is barking.`);
  }
}

let dog = new Dog("Buddy", "Golden Retriever");
dog.eat();    // "Buddy is eating."   ← inherited from Animal
dog.bark();   // "Buddy is barking."  ← own method
super() rules
In a subclass constructor, you must call super() before accessing this. If you forget, you'll get a ReferenceError.

ES6 Module System

ES6+

Each file is its own module with its own scope. Nothing is shared by default — you must explicitly export and import.

  • JS originally had no module system — several competing systems emerged (CommonJS, AMD, UMD). Modern JS has standardized on ES6 modules.
  • Like Python, each file = one module.

Exporting

Export individually
export const PI = 3.14159;

export function add(a, b) {
  return a + b;
}
Export all together (with alias)
const PI = 3.14159;
function add(a, b) { … }

export {
  PI,
  add as addition
};
Default export (one per file)
export default class Point {
  constructor(x, y) { … }
}
Re-export from another module
export { add } from "./foo.js";
export * from "./utils.js";

Importing

// Named import — curly braces required
import { add } from "./foo.js";

// Multiple named imports
import { PI, add } from "./foo.js";

// Named import with alias
import { add as myAdd } from "./foo.js";

// Import everything into a namespace object
import * as foo from "./foo.js";
foo.add(2, 3);

// Default import — NO curly braces
import Point from "./Point.js";
Common Gotcha
Named exports require { } when importing. Default exports do not. Mixing them up is one of the most frequent ES6 module errors.
