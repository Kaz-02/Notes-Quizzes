01

The Problem — Int Stack

Java

Imagine a stack that works for int. The exact same logic would be needed for String, Double, or any other type. Writing a separate class for every type is impractical.

public class Stack {
  private int[] elements;
  // ...
  public void push(int x) { … }
  public int  pop()       { … }
  public int  peek()      { … }
}
The Problem
This class only works for int. Duplicating it for every data type violates DRY (Don't Repeat Yourself) and is a maintenance nightmare.
02

Object Stack & Its Drawbacks

Java

One approach: use Object as the type — Java's root superclass. Polymorphism lets any object be stored.

public class Stack {
  private Object[] elements;
  // ...
  public void push(Object x) { … }
  public Object pop()        { … }
  public Object peek()       { … }
}
Benefit

Works for any type. No code duplication needed.

Drawbacks

No type safety — anything can be pushed, even if mixed types are a bug.

Manual casting required — every pop() returns an Object that must be cast back, risking ClassCastException at runtime.

03

Types of Polymorphism

concept

There are three fundamental kinds of polymorphism in programming languages:

Ad-hoc
a.k.a. Overloading
Same function name, different parameter types. Resolved at compile time.
Subtype
Inheritance / Interfaces
A subtype can be used wherever a supertype is expected. Resolved at runtime.
Parametric
a.k.a. Generics
Code written once works for many types, specified as type parameters.
Key Idea
Generics (parametric polymorphism) solve the Stack problem: write the code once, substitute the type later — with full type safety.
04

Generic Stack

Java

Declare a type parameter <T> on the class. T is a placeholder that gets substituted with a real type when the class is instantiated.

public class Stack<T> {        // T is the type parameter
  private T[] elements;
  // ...
  public void push(T x)  { … }
  public  T   pop()        { … }
  public  T   peek()       { … }
}

// Instantiate with a concrete type — no casting needed
var stack = new Stack<String>();
stack.push("hello");
String s = stack.pop();   // returns String, no cast!
Benefit over Object Stack
Type errors are caught at compile time instead of causing ClassCastException at runtime. One class definition works for any type.
05

More Java Generic Examples

Java

Java's java.util package is built on generics. You use them every day:

// Generic class with one type param
List<String>    names   = new ArrayList<>();

// Generic class with TWO type params
Map<String, Integer> scores = new HashMap<>();

Generic Methods — the type parameter is declared before the return type:

public class Utility {
  public static <T> void printArray(T[] array) {
    // ↑ type param declared BEFORE return type (void)
    for (T element : array) {
      System.out.println(element);
    }
  }
}

// Calling — Java infers T from the argument, no <T> needed
String[] words = { "hello", "world" };
Utility.printArray(words);   // T inferred as String
06

Constraints on Type Parameters

Java

Sometimes an unbounded T is too permissive. Consider findMax — it needs to compare elements, but a plain T has no guarantee of supporting comparison:

// Problem: T could be anything — how do we compare?
public static <T> T findMax(T[] array) {
  // Can't call array[i].compareTo(array[j]) — T might not have it!
}
The Issue
An unconstrained T can be any type. You cannot call methods on it that aren't defined on Object. You need a bounded type parameter.
07

Bounded Type Parameter

Java

Use extends to constrain T to types that implement a specific interface or extend a specific class. This guarantees the required methods exist.

// T must implement Comparable<T> — guarantees compareTo()
public static <T extends Comparable<T>> T findMax(T[] array) {
  T max = array[0];
  for (T item : array) {
    if (item.compareTo(max) > 0) max = item;
  }
  return max;
}
  • Comparable<T> is itself a generic interface — interfaces can be generic too.
  • The keyword is extends even for interfaces (not implements) — this is a Java generics quirk.
  • Multiple bounds: <T extends Comparable<T> & Serializable> — use & to add more.
08

Wildcard Type ?

Java

Wildcards are used when consuming a generic type rather than defining one. When you want a method to accept a List of any type without knowing the specific type at the call site:

// List<?> = a list of some unknown type
public void reverse(List<?> list) {
  // Can read elements as Object, but cannot add non-null elements
  Collections.reverse(list);
}

// Accepts any of these without complaint:
reverse(new ArrayList<String>());
reverse(new ArrayList<Integer>());
reverse(new ArrayList<Dog>());
Wildcard vs. Type Parameter
Use a type parameter <T> when you need to refer to the type elsewhere in the signature (e.g., return type or another parameter). Use a wildcard <?> when you don't — it signals "I don't care what the exact type is."
09

Variance

concept

Variance describes how the subtyping relationship between complex types (like generics) relates to the subtyping of their component types. Given that Dog is a subtype of Animal, what is the relationship between List<Dog> and List<Animal>?

Covariant
List<Animal> ↑ subtypes List<Dog>

Subtype goes in the same direction

Contravariant
List<Dog> ↑ subtypes List<Animal>

Subtype goes in the opposite direction

Invariant
List<Animal> ✕ no relation List<Dog>

No subtype relationship at all

Why it matters
Getting variance wrong causes either type errors (too strict) or runtime crashes (too permissive). The next slides show why List<T> can't safely be covariant or contravariant when it's both readable and writable.

The core rule (derived from the analysis):

Usagea.k.a.VarianceWhy
Read-onlyProducerCovariantYou only read — a List<Dog> can safely be read as a List<Animal> since every Dog is an Animal.
Write-onlyConsumerContravariantYou only write — a List<Animal> can safely accept Dogs written to it.
Read-writeBothInvariantDoing both opens holes in type safety — must be invariant.
10

Java Invariance & PECS

PECSJava

All Java generics are invariant by default — List<Dog> and List<Animal> have no subtype relationship, even though Dog extends Animal.

// Both are TYPE ERRORS in Java:
Iterator<Dog>     d1 = …;
Iterator<Animal>  a1 = d1;   // ✕ compile error

Comparator<Animal> a2 = …;
Comparator<Dog>    d2 = a2;   // ✕ compile error

Java's solution is use-site variance via bounded wildcards, captured in the PECS principle:

Producer Extends,   Consumer Super Use <? extends T> for covariance (reading) · Use <? super T> for contravariance (writing) 
// Iterator only reads (produces values) → covariant → extends
Iterator<Dog>               d1 = …;
Iterator<? extends Animal>  a1 = d1;   // ✓ OK — ? extends Animal covers Dog

// Comparator only writes/consumes → contravariant → super
Comparator<Animal>           a2 = …;
Comparator<? super Dog>      d2 = a2;   // ✓ OK — ? super Dog covers Animal
Memory trick
If you're getting items out of a structure — use extends (producer). If you're putting items in — use super (consumer). If you're doing both — use neither (invariant).
11

Raw Types

Java

Before Java 5 added generics, all collections used Object. These are called raw types — effectively a generic type with T = Object.

Generic TypeRaw TypeEquivalent to
List<T>ListList<Object>
ArrayList<T>ArrayListArrayList<Object>
Map<K,V>MapMap<Object,Object>
  • Raw types exist for backward compatibility — pre-Java 5 code had no generics.
  • Modern Java code should never use raw types — they bypass the type system and suppress compiler warnings.
  • Using a raw type generates an "unchecked" compiler warning.
12

Type Erasure

Java

Java generics are implemented via type erasure: the compiler enforces type safety at compile time, then removes all generic type information from the bytecode. At runtime, ArrayList<String> and ArrayList<Integer> are both just ArrayList.

Compile time ArrayList<String>
→
Runtime (erased) ArrayList

This is why certain things that seem logical do not compile:

// Cannot create instance of T — type is unknown at runtime
T t = new T();              // ✕ compile error

// Cannot create array of T — JVM needs concrete type for array
T[] elements = new T[capacity];  // ✕ compile error

// Cannot use instanceof with generic type
if (obj instanceof List<String>) { … }  // ✕ compile error

// Workaround — pass Class<T> token at construction time
public class Foo<T> {
  private Class<T> type;
  public Foo(Class<T> type) { this.type = type; }
}
Why erasure?
Java chose type erasure (rather than reification, where types are kept at runtime) to remain backward compatible with pre-Java-5 code. Languages like C# retain generic type info at runtime (reified generics), avoiding these limitations.
