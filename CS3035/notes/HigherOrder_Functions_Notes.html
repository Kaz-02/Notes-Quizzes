01

Higher-Order Functions

Definition

A higher-order function is any function that either takes other functions as parameters, returns a function as its result, or both. This is possible because functions are first-class citizens in JavaScript (and Python, etc.).

JavaScript Example — withLogging decorator

This function wraps any other function and logs its arguments and return value. It accepts a function and returns a new function — both criteria for a HOF.

JS
function withLogging(fn) {            // takes a function as param
  return function (...args) {         // returns a new function
    console.log("Calling with args:", args);
    const result = fn(...args);       // ...args spreads the array back into args
    console.log("Result:", result);
    return result;
  };
}

// Usage
function add(a, b) { return a + b; }
const loggedAdd = withLogging(add);
loggedAdd(2, 3);
// Calling with args: [2, 3]
// Result: 5
Use typeof fn === "function" to check at runtime whether a parameter is actually a function before calling it.
Python Equivalent — @decorator syntax
Python
def with_logging(fn):
    def log(*args):
        print("Calling with args:", args)
        result = fn(*args)
        print("Result:", result)
        return result
    return log

# @decorator is shorthand for: add = with_logging(add)
@with_logging
def add(a, b):
    return a + b
C — Function Pointer

Pass a function's address as a raw pointer. No type safety — any pointer signature can be assigned, causing runtime crashes (e.g. passing void print() where int(int,int) is expected).

C
int (*funcPtr)(int, int);
funcPtr = add;
int result = funcPtr(3, 4);
C# — Delegate

A delegate is a typed function reference. The compiler enforces that only functions matching the declared signature can be assigned — much safer than C pointers.

C#
public delegate int MathOperation(int a, int b);
MathOperation op = Add;
var result = op(1, 2);
JavaScript is dynamically typed — functions have no enforced type signature, so any function can be passed to any HOF. Python works the same way. C# delegates enforce type at compile time.
02

Function Types, Subtyping & Variance

Subtype Polymorphism — The Basics

A variable of type Animal can hold a Dog (subtype), but not the reverse. A Dog variable cannot hold a plain Animal because the animal may lack dog-specific features.

Java / C#
Animal a = new Dog();    // ✔ Dog IS-A Animal
Dog    d = new Animal(); // ✘ Animal is NOT necessarily a Dog
Function Type Variance — Which functions fit delegate Dog DogOp(Dog d)?

Type hierarchy: Animal → Dog → Pug (Pug is most specific, Animal is most general)

FunctionSignatureCompatible?Reason
Func1 Animal(Dog x) ✔ Yes Return Animal ⊇ Dog (covariant OK). Param Dog = Dog.
Func2 Pug(Dog x) ✔ Yes Return Pug ⊆ Dog (covariant). Param exact match.
Func3 Dog(Animal x) ✔ Yes Return exact. Param Animal ⊇ Dog (contravariant OK).
Func4 Dog(Pug x) ✘ No Param Pug ⊆ Dog — caller may pass a non-Pug Dog. Unsafe.
Func5 Animal(Pug x) ✘ No Same param problem as Func4.
Func6 Pug(Animal x) ✔ Yes Param contravariant ✔, return covariant ✔.
Return type — Covariant

The return type of the assigned function can be the same or a subtype. A caller expecting Dog is happy receiving a Pug (a more specific dog).

Parameter type — Contravariant

The parameter type can be the same or a supertype. A function that accepts any Animal can safely handle whatever Dog the caller passes in.

03

Binding Policy

The Binding Problem — What does a free variable refer to?

When a function references a variable not in its own scope, which binding of that variable does it use? This only matters under dynamic scope.

Pseudo-code (dynamic scope)
int x = 1;               // global x

int foo(int (*bar)()) {
  int x = 3;             // foo's local x
  return bar();
}

int baz() {
  return x;              // which x?
}

void main() {
  int x = 4;             // main's local x
  print(foo(baz));       // passes baz as argument to foo
}
Deep Binding

Use the environment where the function is referenced / passed.

Here: baz is referenced in main() → uses main's x = 4.

Output: 4

This is the behavior of static (lexical) scope — JavaScript uses this.

Shallow Binding

Use the environment where the function is called.

Here: baz is called inside foo() → uses foo's x = 3.

Output: 3

Used in some dynamic scoping languages. Less predictable.

JavaScript uses static (lexical) scope with deep binding. A function always sees the variables from where it was defined, regardless of where it's called. This is what makes closures work reliably.
04

Closures

Definition

A closure is a function bundled together with its surrounding lexical environment. When an inner function references variables from an outer function, those variables are "captured" and kept alive even after the outer function has returned.

Function + Captured Variables = Closure
Closure Example — Independent State per Call
JS
function foo() {
  let value = 10;                   // captured variable
  return () => { console.log(value++); };  // inner arrow fn closes over value
}

let bar1 = foo();   // bar1 gets its OWN copy of value=10
let bar2 = foo();   // bar2 gets a SEPARATE copy of value=10

bar1();   // 10  (value becomes 11)
bar1();   // 11  (value becomes 12)
bar2();   // 10  (bar2's value is independent)
bar2();   // 11
Console output
bar1(): 10
bar1(): 11
bar2(): 10
bar2(): 11
Each call to foo() creates a new closure with its own independent value variable. bar1 and bar2 do not share state. This is the key insight — closures give functions private, persistent memory.
Common Use Cases

Factory functions, partial application / currying, private state (module pattern), event handlers that remember context, memoization.

Common Gotcha

Using var in loops — all closures share the same variable (declared once, function-scoped). Fix: use let (block-scoped) or wrap in an IIFE.

Functors & Operator Overloading

What is a Functor?

A functor is an object that can be used syntactically like a function. In C++, this is achieved by overloading the function call operator operator().

The key advantage over plain functions: a functor is an object, so it can maintain internal state that persists across calls — similar to a closure in JavaScript.

Operator Overloading — Point + Point
C++
class Point {
public:
  double x, y;
  Point operator+(Point other) {
    return Point(
      x + other.x,
      y + other.y
    );
  }
};
// Usage:
Point p = p1 + p2;
Functor — Counter with State
C++
class Counter {
public:
  int count = 0;
  // overload () operator
  int operator()() {
    return count++;
  }
};
Counter counter;
counter();  // returns 0
counter();  // returns 1
Functor vs Closure — Parallel Comparison
ConceptLanguagePersistent StateMechanism
Closure JavaScript / Python ✔ Yes (captured vars) Inner function captures outer scope variables
Functor C++ / Java ✔ Yes (object fields) Object with overloaded operator()
Function Pointer C ✘ No Raw pointer — no environment bundled
Delegate C# ✔ Via lambda Typed function reference; lambda captures context
The unifying idea: all of these mechanisms let you pass behavior as a value. Closures and functors add the ability to carry state with that behavior, enabling powerful patterns like decorators, factories, and stateful callbacks.
