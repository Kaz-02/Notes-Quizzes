Type & Typing

Study notes compiled from lecture slides

01

Why Type Matters

  • Determines how data is stored and accessed in memory
  • Prevents common errors — catches type mismatches before they cause bugs
  • Improves readability and documentation — types are self-documenting
  • Better maintainability and scalability of large codebases
  • Enables better tooling: autocomplete, refactoring, static analysis
The rise of TypeScript (JavaScript + static typing) is a real-world proof that developers value type safety even in dynamic environments.
02

What is Typing?

Typing describes how a language classifies values into types and how it checks and enforces rules about those types.

03

Primitive and Composite Types

Primitive types are basic, built-in types defined by the language itself — e.g., int, char, bool, float.

Composite types are constructed from primitive or other composite types — e.g., array, list, struct, class.

Function types treat functions as first-class values. More on this when we cover functional programming.

# Python type hint examples
def add(a: int, b: int) -> int:
    return a + b

numbers:     list[int]         = [1, 2, 3]
coordinates: tuple[float, float] = (10.5, 20.3)
user_ages:   dict[str, int]    = {"Alice": 25}
04

Type Equivalence

When are two types considered the same?

  • Structural equivalence — two types are equal if they have the same internal structure
  • Name equivalence — types are equal only if they share the same declared name (used in most modern languages)
struct foo { int a; float b; };
struct bar { int a; float b; };

// Name equivalence  → foo ≠ bar  (different names)
// Structural equiv. → foo = bar  (same structure)
C/C++ uses name equivalence — foo and bar are treated as distinct types despite having identical fields.
05

Type Compatibility & Coercion

  • Type Coercion — automatic or explicit conversion between compatible types
  • Polymorphism (Subtype) — a subtype can be used wherever its supertype is expected
double x = 10;             // int → double  (implicit, widening)
int    y = (int) 10.1;     // double → int  (explicit, narrowing / cast)
List   z = new ArrayList(); // subtype polymorphism (Java)
KindDirectionHow
ImplicitWidening (safe)Done automatically by the language
Explicit (cast)Narrowing (lossy)Programmer must write the cast explicitly
06

Supertype and Subtype

  • Superclass / subclass — via class inheritance
  • Super-interface / sub-interface — via interface hierarchy
  • Structural typing — compatibility based on shape, not declared name
// TypeScript — structural typing
type Animal = { name: string }
type Dog    = { name: string; breed: string }

// Dog has all properties of Animal
// → Dog is a subtype of Animal (no explicit declaration needed)
const greet = (a: Animal) => console.log(a.name);
const myDog: Dog = { name: "Rex", breed: "Husky" };
greet(myDog); // ✅ valid
07

Strong and Weak Typing

Strong Typing

Fewer implicit coercions. Type errors are caught and reported.

JavaPythonKotlin
Weak Typing

More implicit coercions. The language silently converts types.

JavaScriptPHP
// JavaScript — weak typing example
"5" - 1   // → 4   (string silently converted to number)
"5" + 1   // → "51" (number silently converted to string!)
Important: Dynamic ≠ Weak. Python is dynamically typed but strongly typed — it raises a TypeError rather than silently coercing mismatched types.
08

Duck Typing

"If it walks like a duck and quacks like a duck, it's a duck."
  • Whether an object can be used for a task is determined by the presence of certain methods/properties, not its declared type
  • A form of dynamic typing
class Dog:
    def speak(self): print("Woof")

class Person:
    def speak(self): print("Hello")

def make_it_speak(thing):
    thing.speak()   # works for anything that has speak()

make_it_speak(Dog())     # → Woof
make_it_speak(Person())  # → Hello
In Java, you'd need to define a Speakable interface explicitly. Duck typing is flexible, but type errors are only caught at runtime.
09

Type Inference

The compiler/runtime automatically deduces the type of a variable from its context — no need to annotate everything.

var a = 10;                            // int
var b = "hello " + a * 2.2;             // String (arithmetic → concat)
var c;                                 // ❌ error — no initializer
var d = x ? new Dog() : new Animal(); // Animal (common supertype)
var e = y ? "abc" : 12;               // ❌ error / any — incompatible types
Type inference reduces boilerplate while preserving static type safety. Ambiguous cases may produce errors or fall back to a top type (any / Object).
