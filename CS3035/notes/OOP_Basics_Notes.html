01

Why OOP?

Object-Oriented Programming gives us two major advantages over procedural code:

Benefit 01
Real-world modeling

Real-world "things" map naturally to classes and objects, making code easier to reason about.

Benefit 02
Modularity & organization

Better code organization leads to higher scalability and maintainability over time.

02

OOP Terminology

Term
Class

A blueprint or template that defines the structure and behavior of objects.

Term
Object / Instance

A concrete realization of a class. Each object has its own copy of instance fields.

Term
Field / Attribute

Data stored in a class or object. Also called instance variable, data member.

Term
Method

A function defined inside a class. Also called member function.

Term
static

Associated with the class rather than any specific object/instance.

Example — BankAccount class:

Java 
public class BankAccount {
  public static double interestRate = 0.005; // class variable (shared)
  private String owner;                       // instance variable
  private double balance;                     // instance variable

  public BankAccount(String owner, double balance) { … }
  public double balance() { … }
  public void deposit(double amount) { … }
  public void withdraw(double amount) { … }
}
03

this and self

Both keywords refer to the current object on which the method is being called. The key difference is how they're passed:

Java
this (implicit)

Passed automatically behind the scenes. You write this.field to disambiguate from parameters.

Python
self (explicit)

Must be declared as the first parameter of every instance method. Python passes it automatically when you call the method.

Java — disambiguating with this 
public void setName(String name) {
  this.name = name; // "this.name" = field, "name" = parameter
}
Python — explicit self 
def set_name(self, name):
  self.name = name
04

Accessing Class & Object Members

In Java, static members belong to the class; instance members belong to an object. Mixing them up causes compiler errors.

Java — access patterns 
public class Test {
  public static int a = 10; // static (class) field
  public int        b = 20; // instance field

  public static void main(String[] args) {
    var x = a;              // ✅ static field accessed directly
    x = b;                  // ❌ ERROR — can't access instance field without object
    x = Test.a;            // ✅ class.staticField
    x = Test.b;            // ❌ ERROR — Test.b is not static
    x = (new Test()).a;    // ✅ works but unconventional for static
    x = (new Test()).b;    // ✅ instance field via object
  }
}
Key rule: You cannot reference an instance field or method from a static context without first creating an object.
05

Static & Class Methods (Python)

Python decorators distinguish three method types. These decorators are a form of metaprogramming.

@staticmethod
Static Method

No self or cls. Cannot access class or instance data. Used for utility functions grouped inside a class (e.g. Math.sqrt()).

@classmethod
Class Method

Receives cls (current class) as first arg. Can access class-level data. Useful for alternative constructors.

no decorator
Instance Method

Receives self. Full access to both instance and class data. The default.

Python 
class Baz:
  @staticmethod
  def static_name():
    return 'Baz'            # no access to class or instance

  @classmethod
  def class_name(cls):
    return cls.__name__     # receives the class itself
06

Property (Python)

A property is defined by methods but accessed like a regular field. It's backed by an internal field (typically prefixed with _) that shouldn't be accessed directly.

Python — Circle with area property 
import math

class Circle:
  def __init__(self, radius):
    self._radius = radius    # internal backing field

  @property
  def area(self):            # getter
    return math.pi * self._radius ** 2

  @area.setter
  def area(self, value):    # setter — derives radius from area
    self._radius = math.sqrt(value / math.pi)

c = Circle(5)
c.area = 100               # looks like a field, calls setter
print(c.area)              # looks like a field, calls getter
Why use it? Properties let you add validation or computed logic without changing the calling syntax.

Access Modifiers (Java)

Java controls visibility of class members through four access levels:

Modifier Accessible from… Example use
public Everywhere API methods users call
private Within the same class only Internal fields, helper methods
(default) Within the same package Package-internal helpers
protected Same package + subclasses Fields intended for inheritance
Java — example 
package cs3035;

public class Foo {
  int x;  // default — accessible within cs3035 package
}

public class Bar {
  int y;
  public void test(Foo f) {
    y = f.x;  // ✅ same package, works
              // ❌ if x were private — compile error
  }
}
08

Nested & Local & Anonymous Classes

Nested Class

  • Defined inside another class
  • Accesses outer class's private members
  • Classic use: iterator classes

Local Class

  • Defined inside a method
  • Scope limited to that method
  • "One-off" class needing method locals

Anonymous Class

  • No name — defined & instantiated at once
  • Implements an interface inline
  • Often replaced by lambdas in modern Java
Java — nested class accessing private field 
class Outer {
  private int x = 10;

  class Inner {
    void show() {
      System.out.println(x);  // ✅ can access private x
    }
  }
}
Java — anonymous class 
interface Greeting {
  void sayHello();
}

Greeting frenchGreeting = new Greeting() {
  public void sayHello() {
    System.out.println("Bonjour!");
  }
};

Encapsulation

Encapsulation means bundling data and the operations on it together, while restricting direct access to the internal state.

Goal 01
Security & integrity

Prevents external code from putting objects into invalid states.

Goal 02
Decoupling

Implementation can change without breaking client code, as long as the public interface stays the same.

Goal 03
Reduced complexity

Users only see what they need. Internal details are hidden.

Golden rule: Make fields private and expose only what's necessary through public methods (getters/setters or higher-level operations).
