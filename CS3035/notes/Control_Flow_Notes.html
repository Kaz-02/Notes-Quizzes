01 Operators & Arity

Arity refers to how many operands an operator takes.

ArityDescriptionExamples
UnaryTakes one operandnot x, -x (note: ++ does NOT exist in Python)
BinaryTakes two operands+, -, *, /
TernaryTakes three operandsThe conditional operator

Python's ternary (conditional) operator differs from Java's ?: syntax:

Java

x = a + b > 10 ? 1 : -1;

Python

x = 1 if a + b > 10 else -1
🧠 Try it Given a = 10; b = 20, trace through: 
x = y = 100 + a * b if not (a + b < 100) else a * b - 100
x += y
print(x)
Step-by-step: a + b = 30, so not (30 < 100) = not True = False → takes the else branch → a * b - 100 = 200 - 100 = 100. So x = y = 100, then x += y → x = 200.
200
02 Precedence

Precedence determines which operator is evaluated first when multiple operators appear in an expression.

Unary ↓ higher priority
Arithmetic
Comparison
Logical
Ternary ↓ lower priority

So in x = a + b > 10 ? 1 : -1, + (arithmetic) runs before > (comparison) which runs before ?: (ternary).

💡 Rule of thumb When in doubt, use parentheses to make the evaluation order explicit and your code more readable.
03 Associativity

When operators have the same precedence, associativity determines evaluation order.

Left-associative (most operators)

  • Evaluated left → right
  • 10 - 5 - 2 = (10 - 5) - 2 = 3

Right-associative (Python)

  • = assignment
  • ** exponentiation
  • Evaluated right → left
🔍 Trace this 
x = 10
y = x = 10 - 5 - 2

Assignment = is right-associative. The RHS 10 - 5 - 2 is left-associative → evaluates to 3. Then x = 3, then y = x = 3.

x = 3, y = 3

Non-Associative operators

Some operators can't be chained — they're non-associative:

Java — Error!

a < b < c  // TypeError

Python — Valid!

a < b < c
# means: a < b and b < c

Python supports chained comparisons natively — a uniquely Python-friendly feature.

04 Short Circuiting

Short circuiting is a compiler/interpreter optimization where evaluation of a boolean expression stops as soon as the final result is determined.

ExpressionStops early when…
A and BA is False — B is never evaluated
A or BA is True — B is never evaluated
🛡️ The Guard Pattern Short circuiting enables a safe access pattern — check for None before accessing attributes: 
if obj is not None and obj.value > 0:
    print(obj.value)
# If obj is None, obj.value is never accessed — no crash!
🧠 Trace this 
def foo(x):
    print(x)
    return x * x

x = 5
if x > 10 and foo(x) < 1000:
    print(x)

x > 10 is 5 > 10 = False. Because of short circuiting, foo(x) is never called. Nothing is printed.

(no output)
05 Compound Assignment

Compound assignment (+=, -=, etc.) is not always equivalent to the expanded form.

FormHow many times x is evaluated
x += 1Once — more efficient
x = x + 1Twice — x appears on both sides
⚠️ Tricky Example 
def foo(values):
    values.append(0)
    return values

mylist = []
foo(mylist)[0] += 1          # foo() called ONCE
foo(mylist)[0] = foo(mylist)[0] + 1  # foo() called TWICE → two 0s appended!

The second form calls foo() twice, so the list gets two zeros appended before any increment happens. The results differ!

💡 Key idea Structured control statements are predictable and readable — they have a clear entry point and exit point. This is why they replaced goto.
07 Unstructured Control — GOTO

The GOTO statement jumps execution to any arbitrary label in the code. It's considered harmful because it makes code hard to read, debug, and reason about.

int x = 0;
HERE: printf("%d\n", x);
x++;
goto HERE;   // jumps back to HERE — infinite loop
📖 Historical Note Edsger Dijkstra's famous 1968 letter "Go To Statement Considered Harmful" argued that overuse of GOTO makes programs difficult to understand and verify. This led to the shift toward structured programming. Python has no GOTO statement.
08 Exceptions

An exception is a mechanism for a program to signal, propagate, and handle errors in exceptional situations — outside the normal flow of execution.

Python

try:
    x = int(s)
    print(x)
except ValueError:
    print("Invalid number")

Java

try {
  int x = Integer.parseInt(s);
  System.out.println(x);
} catch (NumberFormatException e) {
  System.out.println("Invalid");
}

Why exceptions? — Separating error handling from normal logic

❌ Without exceptions (nested error codes)

if fileIsOpen:
  if gotLength:
    if gotMemory:
      if not readFailed:
        ...
      else: errorCode = -1
    else: errorCode = -2
  else: errorCode = -3
else: errorCode = -5

✅ With exceptions (clean separation)

try:
  open the file
  determine its size
  allocate memory
  read file into memory
  close the file
except FileOpenFailed: ...
except MemoryError: ...
except ReadError: ...

Error propagation with exceptions

def divide(a, b):
    return a / b               # raises ZeroDivisionError if b == 0

def compute():
    return divide(10, 0)       # doesn't need to handle it

def main():
    try:
        result = compute()
        print(result)
    except ZeroDivisionError:
        print("Cannot divide by zero")  # handled here, once
✅ With exceptions Errors propagate automatically up the call stack and are handled at the level of your choosing — no manual error code passing at every function level.
⚠️ Know the difference "Cannot connect to database" → exception (unexpected, hardware-level failure)
"Wrong password entered" → normal error (expected user mistake, handle with if/else)
09 Best Practices — Less Control Flow is Often Better

Every branch and loop you add has a cost:

  • Repetition is expensive — loops consume CPU cycles
  • Branching stalls CPU pipelines — the processor has to guess which branch to take; wrong guesses waste cycles
  • More branches = more cognitive load — harder to read, test, and maintain

Eliminating branches with a lookup table

❌ With if-elif chain

def weekday(n):
    if n == 1: return "Monday"
    elif n == 2: return "Tuesday"
    elif n == 3: return "Wednesday"
    ...7 branches total

✅ With a list (no branches)

days = ["Monday", "Tuesday",
        "Wednesday", "Thursday",
        "Friday", "Saturday",
        "Sunday"]

def weekday(n):
    return days[n - 1]

Eliminating loops with a lookup table

❌ With a while loop

def count1s(n):
    """Count 1-bits in 4-bit int"""
    count = 0
    while n:
        count += n & 1
        n >>= 1
    return count

✅ Lookup table (0–15 only)

bits = [bin(i).count('1')
        for i in range(16)]

def count1s(n):
    return bits[n]
# O(1) lookup — no loop at runtime!
💡 Key takeaway Lookup tables (lists, dicts) can replace both branches and loops, turning O(n) operations into O(1) lookups. This is a fundamental optimization pattern in software engineering.
