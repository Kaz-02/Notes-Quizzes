OSが提供するサービス（Layer 1）

USER VIEW▼
ユーザー向けサービス

UI（CLI/GUI/タッチ）、プログラム実行（ロード・実行・終了）、I/O操作、ファイル操作（作成・削除・読み書き・権限）、プロセス間通信（共有メモリ/メッセージパッシング）、エラー検出

システム効率向けサービス

リソース配分（CPU・メモリ・デバイス）、ロギング（使用統計の記録）、保護とセキュリティ（アクセス制御・外部攻撃への防御）

🖥️

ユーザーインターフェースの3種類 ＋ シェルの仕組み

USER VIEW▼
CLI（コマンドライン）

テキストで直接コマンド入力。シェル（bash等）が解釈・実行。パワーユーザーや自動化に最適。

GUI（グラフィカル）

ウィンドウ・アイコン・マウス操作。デスクトップ比喩。Xerox PARCが発明、Appleが普及。

タッチスクリーン

スワイプ・タップ操作。スマートフォン標準。iOS, Android等。

シェルの2つの実装方式
組み込みコマンド方式

シェル自身がコマンドのコードを内包。追加はシェル自体の改変が必要。シェルが大きくなる。

システムプログラム方式（UNIXの方法）

コマンドは独立した実行ファイル。シェルはファイルを検索・実行するだけ。シェルが小さく拡張性◎。

📞

システムコール：カーネルへの唯一の合法的な入り口

PROGRAMMER▼
アプリはハードウェアを直接制御できない。OSサービスを使うには決められた窓口 システムコール を通じてカーネルにリクエストする。単純なプログラムでも毎秒数千回呼び出す。
ユーザーモード（mode bit=1）からカーネルモード（mode bit=0）への唯一の合法的な入り口。
6つのカテゴリー
① プロセス制御（fork, exit, wait） ② ファイル管理（open, read, write, close） ③ デバイス管理（ioctl, read, write） ④ 情報管理（getpid, alarm, sleep） ⑤ 通信（pipe, shm_open, mmap） ⑥ 保護（chmod, umask, chown）
パラメータ渡しの3方法
① レジスタ渡し

最シンプル。CPUレジスタに直接格納。Linuxは5つ以下に使用。数が制限されるのが欠点。

② メモリブロック渡し

パラメータをメモリに格納しアドレスをレジスタに入れる。数・長さに制限なし。

③ スタック渡し

プログラムがPUSH、OSがPOPして取得。同様に数・長さに制限なし。

🔌

API・ABI：抽象化レイヤーと互換性

PROGRAMMER▼
プログラマーは通常 API（高レベルな関数群）を使う。ライブラリ（libc等）がシステムコールへの変換を担う。例：printf() → write() システムコール
API（Application Programming Interface）

ソースコードレベルの互換性。標準API（POSIX等）を使えば再コンパイルだけで別OSでも動く。

主要API：Windows API、POSIX（Unix/Linux/macOS）、Java API（JVM）

ABI（Application Binary Interface）

バイナリレベルの互換性。コンパイル済みコードがそのまま動く。アーキテクチャに依存。

規定するもの：アドレス幅（32/64bit）、システムコールのパラメータ渡し規約、スタック構造、ライブラリのバイナリ形式

なぜアプリはOS固有なのか？ → ポータビリティの解決策

PROGRAMMER▼
固有のシステムコール

各OSは独自のセットを持つ。
Linux：300以上
Windows：約2000
macOS：300〜500

実行ファイル形式の違い

Windows → .exe（PE形式）
Linux → .elf（ELF形式）
macOS → .mach-o（Mach-O）

Mach-Oは「Fat Binary」で複数アーキテクチャを1ファイルに。

ポータビリティの3解決策

① インタープリタ（Python, Ruby）：遅いが簡単
② 仮想マシン（Java/JVM）：バイトコードを中間層で実行
③ ネイティブポーティング（C/POSIX）：高労力

⚙️

黄金律：ポリシーとメカニズムの分離

ARCHITECT▼
メカニズム（How）

「どうやって」実現するか。例：タイマー割り込みを設定する仕組み。カーネルに実装。

ポリシー（What）

「何を」するか。例：特定ユーザーにタイマーを何秒設定するか。変更が容易。

メカニズムが汎用的であれば、カーネルを書き直さずにポリシー（I/OとCPUの優先順位など）だけを変更できる。
🏗️

カーネルアーキテクチャの4パターン

ARCHITECT▼
① モノリシック

全機能を単一アドレス空間に。密結合。例：伝統的UNIX

✓ 高速（内部通信コスト低）
✗ デバッグ・拡張困難。1バグが全体をクラッシュ
② マイクロカーネル

最小限のみカーネルに残し、残りをユーザー空間へ。通信はメッセージパッシング。例：Mach, QNX

✓ 安全・信頼性高い、拡張容易
✗ メッセージパッシングのオーバーヘッドで性能低下
③ モジュラー（LKM）

コアカーネル＋動的にロード可能なモジュール。起動時・実行時に追加・削除。例：Linux, macOS, Windows

✓ 再コンパイル不要で柔軟。モノリシック並みの性能
✗ モジュールがカーネル空間 → 完全隔離はなし
④ ハイブリッド（現実解）

複数のアーキテクチャを組み合わせて性能・安全・柔軟性を最適化。現代のOSはほぼハイブリッド。例：macOS Darwin（Mach + BSD）、Android（Linux + ART）

✓ 各手法の長所を組み合わせられる
🔬

ケーススタディ：macOS（Darwin） と Android スタック

ARCHITECT▼
macOS / iOS — Darwin カーネル

Machマイクロカーネル（スケジューリング・IPC・メモリ管理）＋ BSDモノリシックカーネル（POSIX API・ネットワーク・ファイルシステム）の融合。

性能の解決策：BSDとI/O KitをMachと同じアドレス空間で実行 → ユーザー空間メッセージパッシングなしで高速通信を実現。

Android — レイヤードスタック

Applications（Java）
↓ ART（AOTコンパイルでネイティブコードに変換）
↓ Libraries & Frameworks（WebKit, SQLite, Bionic C等）
↓ HAL（多様なハードウェアを抽象化）
↓ 改造Linuxカーネル（省電力・メモリ制約対応）

OSブートの6ステップ

SYSTEM▼
STEP 01

電源 ON

システムが電力を受ける
STEP 02

ファームウェア起動

UEFI / BIOS — 不揮発性メモリ上のブートプログラムが実行
STEP 03

カーネルをロード

ストレージ上のカーネルを見つけてメモリへ読み込む
STEP 04

カーネル初期化

ハードウェアを初期化し、一時RAMファイルシステム（initramfs）を作成
STEP 05

ルートFSをマウント

永続的なルートファイルシステムをマウント
STEP 06

初期プロセス起動

最初のユーザー空間プロセス（systemd等）が起動。システム稼働開始。
