Chapter 4 — 学習ノート

スレッド：並列処理の最小単位

プロセスを「分割」して並列に動かす仕組みを、基礎から実装まで理解する

💡 この章の核心

過去のCPU高速化は「クロック速度を上げる」ことで実現した。しかし現代はマルチコアの時代。1プロセス内に複数のスレッドを持ち、複数コアで並列実行することが性能向上の鍵となる。
スレッドはプロセスより軽量で、コード・データ・ファイルを共有しながら独立して動く「軽量な実行の流れ」だ。

🧵

スレッドとは何か：プロセスの中の軽量な実行単位

CONCEPT ▼
スレッドは CPUを使う基本単位。1つのプロセス内に複数のスレッドが共存できる。

🏆

マルチスレッドの4つの柱（利点）

BENEFITS ▼
① Responsiveness（応答性）

処理の一部がブロックされても他のスレッドが動き続ける。例：ワードプロセッサで1スレッドがキー入力を処理しながら、別スレッドがスペルチェック。シングルスレッドなら全体が固まる。

② Resource Sharing（リソース共有）

スレッドは親プロセスのメモリ・リソースをデフォルトで共有する。プロセス間通信は明示的なIPC（共有メモリやパイプ等）が必要だが、スレッド間は同じアドレス空間を直接使える。

③ Economy（経済性）

スレッドの生成・コンテキストスイッチはプロセスより低コスト。リソースを共有するため作成に必要な時間・メモリが少なく、スイッチも高速。

④ Scalability（スケーラビリティ）

マルチコアCPUで複数スレッドが本当に同時並列実行できる。シングルスレッドプロセスは何コアあっても1コアしか使えない。

並行性（Concurrency）

複数タスクが進捗できる状態。「多くのことを一度にさばく」。シングルコアでも可能。インターリービング（時分割）で実現。

並列性（Parallelism）

複数タスクが本当に同時実行される状態。「多くのことを一度にやる」。マルチコアが必須。並行なしの並列はない。

並列性の2種類
データ並列性（Data Parallelism）

同じ操作をデータのサブセットに分けて複数コアで実行。例：1億件の配列を4コアで4等分して同じ計算。

タスク並列性（Task Parallelism）

異なる処理を複数コアに振り分けて並列実行。例：Core1がUI処理、Core2がI/O処理を同時に担当。

📐

アムダールの法則：並列化の理論的上限

AMDAHL'S LAW ▼
公式
Speedup ≤ 1 / ( S + (1-S)/N )
S = プログラム中の直列（シリアル）部分の割合
N = 処理コア数
Nが∞に近づくと：スピードアップは 1/S に収束
具体例

S = 25%（直列部分が1/4）の場合：
コアをいくら増やしても最大 4倍（= 1/0.25）にしかならない。
S = 50% → 最大2倍。

重要な教訓

直列部分がボトルネック。並列化の恩恵を最大化するには、直列部分を可能な限り小さくすることが鍵。コア数を増やすだけでは限界がある。

⚠️

マルチコアプログラミングの5つの課題

CHALLENGES ▼
① タスクの識別：並列化できる処理を見つける ② バランス：各スレッドに均等な量の仕事を割り当てる ③ データ分割：データを複数コアに分散させる方法 ④ データ依存性：順序依存のある処理を同期させる ⑤ テスト・デバッグ：実行パスが膨大になり再現が困難
🔀

ユーザースレッド vs カーネルスレッド：3つのマッピングモデル

THREADING MODELS ▼
ユーザースレッドがCPU上で実際に動くには、カーネルスレッドへのマッピングが必要。
ユーザースレッド（User-level）

ユーザー空間のスレッドライブラリが管理。切り替えが速くカーネル不介在。しかしカーネルは存在を知らない。例：Javaの旧Green Threads

カーネルスレッド（Kernel-level）

OS（カーネル）が直接管理・スケジューリング。物理プロセッサへの真のマルチタスクをサポート。例：Windows Threads、Linux/POSIX Pthreads

① Many-to-One

多数のユーザースレッド → 1つのカーネルスレッド。管理は効率的。

✓ スレッド管理が高速
✗ 1つがブロックすると全体がブロック
✗ マルチコアで並列実行不可
② One-to-One ★現代の標準

ユーザースレッド1つ → カーネルスレッド1つ。最も広く使われる。例：Linux、Windows

✓ ブロックが全体に波及しない
✓ マルチコアで真の並列実行可能
✗ スレッド作成のたびにカーネルスレッドも作成→オーバーヘッド
③ Many-to-Many

多数のユーザースレッド → 同数以下のカーネルスレッドに多重化。

✓ ブロックが全体に波及しない
✓ 大量のユーザースレッドを柔軟に管理
✗ 実装が複雑
コア数増加に伴い「カーネルスレッド数を制限する必要性」が低下。現代のOSはほぼOne-to-Oneモデル（Linux・Windows）を採用。
📚

スレッドライブラリ「ビッグ3」：明示的スレッド管理

THREAD LIBS ▼
POSIX Pthreads

IEEE 1003.1c仕様。実装ではなく動作の仕様を定義。UNIX/Linux/macOSで標準的。

pthread_create(&tid, &attr, runner, argv[1]);
Windows Threads

カーネルレベルのライブラリ。明示的なハンドルと重いデータ構造。windows.hと直接連携。

CreateThread(NULL, 0, Summation, &Param, 0, &Id);
Java Threads

JVMが管理。ホストOSのスレッドの上で動作。RunnableインターフェースまたはLambdaで実装。

Thread worker = new Thread(new Task()); worker.start();
JVMはホストOSのスレッドライブラリを内部で使用。WindowsではWindows API、Unix/Linux/macOSではPthreadsを利用。
同期 vs 非同期スレッド戦略
同期スレッド（Synchronous）— Fork-Join

親スレッドが子を生成し、全子スレッドの終了を待ってから処理を再開。スレッド間のデータ共有が多い場合に使用。pthread_join() / worker.join()

非同期スレッド（Asynchronous）

親は子を生成後、待たずにすぐ処理を継続。親子が並行して独立に動く。データ共有は少ない。マルチスレッドWebサーバーで典型的に使用。

🤖

暗黙的スレッド（Implicit Threading）：スレッドからタスクへ

IMPLICIT ▼
数百・数千のスレッドを手動管理するのは現実的でない。スレッドの生成・管理をコンパイラとランタイムライブラリに委ねるのが暗黙的スレッド。
旧来方式（明示的）

プログラマーがすべてのスレッドを手動で生成・管理。複雑で安全性確保が困難。

暗黙的スレッド（新方式）

プログラマーはタスク（並列化できる仕事の単位）を特定するだけ。ライブラリがスレッドプールへのマッピングを自動処理。

主な実装技術
スレッドプール

起動時にスレッドを事前生成しておき、タスクが来たら空きスレッドに割り当てる。完了後はプールに戻す。

メリット：毎回生成・破棄のコスト不要。同時スレッド数の上限制御が可能。

Fork-Join（Java）

再帰的な分割統治に最適。問題をTHRESHOLD以下になるまで再帰的にFork→直接解く→Joinで結果を統合。

Work Stealing：暇なスレッドが他スレッドのタスクを盗んで負荷分散。

OpenMP / GCD

OpenMP：#pragma omp parallel for の1行でループを並列化。ランタイムがコア数分スレッドを生成。

Grand Central Dispatch（Apple）：タスクをキューに追加するだけ。シリアル/コンカレントキューを選択。

🛑

スレッドの安全な終了：キャンセレーションの2方式

CANCELLATION ▼
スレッドを完了前に終了させること。例：ブラウザで「停止」ボタンを押す。
① 非同期キャンセレーション ✗ 非推奨

一方のスレッドが対象スレッドを即座に強制終了する。

問題：共有データの更新中や、リソースを保持中に終了させるとデータ破損・リソースリークが発生。

② 遅延キャンセレーション ✓ 推奨

対象スレッドが定期的にフラグをチェックし、自分で安全なタイミングで終了する。

リソースを確実に解放してクリーンに終了できる。

Pthreads：pthread_testcancel()
Java：isInterrupted()

🔒

Thread-Local Storage（TLS）：各スレッドだけのプライベートデータ

TLS ▼
スレッドはデータを共有するが、スレッド固有のデータが必要な場合がある（例：各スレッドのトランザクションID）。
TLSとは

グローバル変数のように見えるが、各スレッドが独自のコピーを持つデータ。スレッドのライフタイム全体で関数をまたいで保持される。

static __thread int threadID;（GCC）

ローカル変数との違い

ローカル変数：1回の関数呼び出し中だけ有効。関数を抜けると消える。

TLS：関数をまたいでスレッドが生きている間ずっと有効。スレッドが終了すると消える。

🔬

ケーススタディ：Windows と Linux のスレッド実装

OS IMPL ▼
Windows — One-to-One モデル

各ユーザースレッドを対応するカーネルスレッドに1:1マッピング。データ構造が明確に分離：

TEB（Thread Environment Block）：ユーザー空間。スレッドID・ユーザーモードスタック・TLS。
ETHREAD（Executive Thread Block）：カーネル空間。親プロセスへのポインタ。
KTHREAD（Kernel Thread Block）：カーネル空間。スケジューリング・同期情報。

Linux — タスク統一モデル

Linuxはプロセスとスレッドを区別しない。すべての実行の流れを「タスク」と呼ぶ。

システムコール clone() でタスクを生成。フラグで挙動を制御：
フラグあり（CLONE_VM等）→ スレッド的動作（VMを共有）
フラグなし → プロセス的動作（VMをコピー）

task_struct が他のデータ構造へのポインタを持ち、共有か独立かをフラグで切り替える柔軟な設計。
