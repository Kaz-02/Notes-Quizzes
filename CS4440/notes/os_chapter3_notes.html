プロセス：コンピューティングの生きた単位

プログラムがどう「生まれ・動き・通信し・死ぬか」を理解する

💡 この章の核心

プログラム（ディスク上の静的なファイル）がプロセス（メモリ上で実行中の動的な実体）になることで、現代のマルチタスキングが成立する。プロセスはただ実行されるだけでなく、生まれ・状態を変え・他のプロセスと通信し・終了するライフサイクルを持つ。

🔄

プログラム vs プロセス：受動から能動へ

CONCEPT ▼
プログラム（受動的）

ディスク上の静的なファイル。命令の羅列。それ自体は何もしない。例：browser.exe

プロセス（能動的）

プログラムをメモリに読み込んだもの。プログラムカウンタ（次の命令アドレス）を持ち、リソースを消費しながら実行する。

1つのプログラムから複数のプロセスを起動できる。例：ブラウザを2つのウィンドウで開くと、同じ実行ファイルから2つの独立したプロセスが動く。
JVMの面白い点
JVM自体も1つのプロセス。その中でJavaプログラムが動く。プロセスは別のプロセスの実行環境になれる（入れ子構造）。
🗂️

プロセスのメモリレイアウト：4つのセクション

MEMORY ▼

Text Section（コード）
Text（テキスト）

実行可能なコード本体。固定サイズ。実行ファイルから読み込む。

Data（データ）

グローバル変数。固定サイズ。初期化済みと未初期化で分かれる（Cの場合）。

Heap（ヒープ）

実行時に動的確保されるメモリ（malloc/new）。上向きに伸びる。

Stack（スタック）

関数呼び出し・ローカル変数・戻りアドレスの一時保存。下向きに伸びる。

OSはStackとHeapが衝突しないよう監視しなければならない。衝突すればスタックオーバーフロー。

スケジューラとコンテキストスイッチ：マルチタスキングのコスト

SCHEDULER ▼
Ready Queue と Wait Queue

Ready Queue：CPU待ちのプロセスが並ぶ。スケジューラがここから選んでDispatch。

Wait Queue：I/O待ちやイベント待ちのプロセスが待機。完了するとReady Queueに戻る。

コンテキストスイッチとは？

CPUを別プロセスに切り替えるとき、現プロセスの状態をPCBに保存し、次プロセスの状態をPCBから復元する操作。

純粋なオーバーヘッド。切り替え中はCPUがユーザーの仕事をしない。典型的には数マイクロ秒かかる。

オーバーヘッドの要因：メモリ速度・レジスタ数・特殊命令の有無 切り替えが多すぎるとスラッシング（処理なしにスイッチだけ）
🌱

プロセスの生成：fork・exec・CreateProcess

CREATION ▼
プロセスは必ず別のプロセスから生まれる。Linuxは systemd（PID=1）を祖先とするツリー構造。
UNIX方式：Clone and Replace

① fork()：親プロセスをほぼ完全にコピーして子プロセスを作成。子にはPID=0が返る。親には子のPIDが返る。

② exec()：子プロセスのメモリ空間を新しいプログラムで置き換える。

この「クローンして置き換え」パターンが、起動前に親子間で通信や設定ができる柔軟性を生む。

Windows方式：Direct Creation

CreateProcess()：子プロセスを作成すると同時にプログラムを読み込む。クローンは行わない。

より直接的なアプローチ。引数で実行ファイル名・セキュリティ設定・継承フラグ・環境変数などを一度に指定する。

💀

プロセスの終了：Zombie と Orphan

TERMINATION ▼
正常終了

子が exit() → 親が wait() で終了ステータスを回収 → OSがプロセスを完全に削除。

ゾンビプロセス

子はすでに終了しているが、親がまだ wait() を呼んでいない状態。PCBのエントリがプロセステーブルに残り続ける。

オーファン（孤児）プロセス

親が wait() せずに先に終了した場合、子プロセスが孤立。init/systemd が自動的に養子縁組して回収する。

pid = wait(&status); — 親がこれを呼ぶことでゾンビを解放し、プロセステーブルのエントリが消える。

モバイル環境での生存競争：Androidの優先度ヒエラルキー

MOBILE ▼
バッテリーとメモリの制約から、AndroidはOOMキラー（Out-Of-Memory killer）が低優先度プロセスを強制終了してリソースを回収する。
① フォアグラウンドプロセス最後に終了 / Critical
② ビジブルプロセス画面に見えているが非フォーカス
③ サービスプロセスBGで音楽再生など
④ バックグラウンドプロセスユーザーには見えない
⑤ エンプティプロセスキャッシュのみ / 最初に終了
iOSは歴史的にすべてのBGアプリを一時停止（suspend） 現代iOSは特定APIでのみ限定的なBGタスクを許可

プロセス間通信（IPC）：2つの根本モデル

IPC ▼
① 共有メモリ（Shared Memory）

OSが共有メモリ領域を確立後、プロセスが直接読み書きしてデータ交換。カーネル不介入で超高速。

POSIX API：shm_open() → ftruncate() → mmap()

典型例：Producer-Consumer問題（有界バッファ）

🟩 比喩：会議室のホワイトボード（誰でも直接書ける）
② メッセージパッシング（Message Passing）

カーネルの send() / receive() を通じてメッセージを交換。共有アドレス空間不要。分散システムに最適。

ネーミング：直接（相手を明示）vs 間接（メールボックス経由）

同期：ブロッキング（同期）vs ノンブロッキング（非同期）

📨 比喩：郵便局経由の手紙（カーネルが仲介）
🔧

パイプ：UNIXの哲学を体現するIPC

PIPES ▼
通常パイプ（Ordinary Pipe）

単方向通信。親子関係が必要。プロセスが終了すると消滅。
出力側（write-end）→ 入力側（read-end）の一方通行。

名前付きパイプ（Named Pipe / FIFO）

双方向（半二重）通信可能。親子関係不要。ファイルシステム上に特殊ファイルとして存在し、明示的に削除するまで残る。

パイプはUNIXコマンドライン哲学の基盤。ls -l | grep 'Aug' の | がパイプ。ls の出力が grep の入力に直結する。
🌐

ネットワーク越しのIPC：ソケットと RPC

NETWORK ▼
ソケット（Socket）

IPアドレス＋ポート番号で一意に識別される通信エンドポイント。ファイルハンドルのように read()/write() でデータを送受信。

ウェルノウンポート（1024以下）：SSH=22, FTP=21, HTTP=80
比喩：建物のアドレス（IP）＋部屋番号（ポート）

RPC（Remote Procedure Call）

ネットワーク越しの関数呼び出しをローカル呼び出しのように見せる高レベル抽象化。ネットワーク通信の複雑さを「スタブ」が隠蔽。

パラメータマーシャリング：マシンアーキテクチャ間（big-endian vs little-endian等）のデータ形式差異を標準フォーマットに変換。

🔬

ケーススタディ：Chromeのマルチプロセスアーキテクチャ

CASE STUDY ▼
Chromeは「なぜタブが1つクラッシュしてもブラウザ全体が落ちないのか」の完璧な例。
Browser Process（1つ）

UI・ディスク・ネットワークI/Oを管理。ブラウザ全体の司令塔。

Renderer Process（タブごと）

HTML/JSをレンダリング。サンドボックス化。1タブ＝1プロセスのため、クラッシュが隔離される。

Plug-in Process（プラグインごと）

Flash等のプラグインを専用プロセスで隔離。バグがあっても他に影響しない。

「悲しいタブ」のアイコンは、プロセス隔離の勝利。1タブのクラッシュが連鎖しない。
